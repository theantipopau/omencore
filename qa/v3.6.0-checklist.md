# OmenCore v3.6.0 Release QA Checklist

Use this checklist to validate release readiness for the v3.6.0 cut.

The 3.6.0 release theme is lightweight resource usage without weakening active fan, tuning, RGB, diagnostics, or Linux reliability. Do not mark resource wins as accepted unless the before/after evidence is recorded.

## 1) Preflight

- [ ] `dotnet restore OmenCore.sln` succeeds on the release machine
- [ ] `dotnet build OmenCore.sln -c Release --no-restore` succeeds after restore
- [ ] `dotnet test OmenCore.sln --no-restore` passes
- [ ] Windows app Release build succeeds
- [ ] Linux CLI build succeeds after restore
- [ ] `VERSION.txt` matches target release version
- [ ] `CHANGELOG.md` and `docs/CHANGELOG_v3.6.0.md` are aligned
- [ ] `docs/ROADMAP_v3.6.0.md` has completed items marked `[X]` and partial work marked `[~]`

## 2) Resource Measurement Gate

- [ ] `resource-footprint.txt` is present in a diagnostics export for each measured scenario
- [ ] Cold startup to first usable UI is timed and recorded
- [ ] Visible Dashboard idle CPU/RAM is recorded
- [ ] Minimized tray-only idle CPU/RAM is recorded
- [ ] OSD-visible CPU/RAM and cadence behavior are recorded
- [ ] Fan curve or fan hold active CPU/RAM and cadence behavior are recorded
- [ ] Lighting page first-open CPU/RAM/provider-load impact is recorded
- [ ] Tuning or OMEN page first-open CPU/RAM/provider-load impact is recorded
- [ ] Before/after numbers are attached to the release evidence or changelog validation notes

## 3) Tray and Cadence Guardrails

- [ ] Tray-only low-overhead mode reaches the 10s ultra-low cadence when no blockers are active
- [ ] Active fan curve or fan hold prevents ultra-low cadence while OmenCore owns fan state
- [ ] OSD-visible state forces responsive cadence even when the main window is hidden
- [ ] Settings live cadence card shows tier, reason, and blockers correctly
- [ ] Diagnostics export cadence reason matches the visible Settings cadence state

## 4) Startup and Provider Laziness

- [ ] Tray startup does not force Dashboard, System Control, Fan Control, RGB, or tuning providers to load
- [ ] Dashboard and General pages do not construct System Control as a side effect
- [ ] RGB providers initialize only when Lighting is opened or an explicit lighting action is used
- [ ] Conflict/tuning scans start only from Monitoring, OMEN, Tuning, or Optimizer surfaces
- [ ] Memory Optimizer visible refresh pauses when the Memory tab is hidden

## 5) Fan and Performance Reliability

- [ ] Auto handoff preserves explicit Auto curve payloads after Max or policy transitions
- [ ] Custom fan curves ramp down or recover from accepted-zero-RPM writes without manual curve switching
- [ ] Max/manual/custom `100%` fan requests report firmware/backend clamp behavior clearly
- [ ] Game profile fan/performance actions apply before Fan Control/System Control pages are opened
- [ ] Fan calibration completion, cancellation, and failure all restore BIOS auto fan control

## 6) RGB and Lighting

- [ ] Lighting ownership header reports HP keyboard backend and provider status
- [ ] OMEN Light Studio/Gaming Hub conflict warning appears when those processes are detected
- [ ] Restore Keyboard action reapplies saved HP keyboard colors through the active backend
- [ ] `rgb-control-path.txt` is present in diagnostics export without forcing lazy providers to initialize

## 7) Memory Optimizer

- [ ] Auto Clean minimum-gap override persists and `0` keeps profile-default cooldown behavior
- [ ] Game-aware quiet-window toggle persists and changes foreground-game cleanup behavior
- [ ] Fullscreen and borderless foreground games use the working-set-only path unless pressure is critical
- [ ] Cleanup result summary includes physical, standby, cache, commit, page file, and modified-page deltas
- [ ] Top-process rows can add working-set exclusions without duplicate entries

## 8) Model Identity and Tuning Safety

- [ ] Exact ProductId capability matches win over broad model-name fallbacks
- [ ] Ambiguous or missing ProductId fallbacks keep risky direct EC writes disabled
- [ ] Model identity diagnostics separate baseboard ProductId from HP support SKU/product number
- [ ] `tuning-safety.txt` is present in diagnostics export without waking tuning providers
- [ ] Startup restore gates and pending-test recovery states are visible in tuning diagnostics

## 9) Linux Restored Build and Runtime Gate

- [ ] Restored full solution build no longer fails due to Linux project assets missing `net8.0/win-x64`
- [ ] `dotnet build src\OmenCore.Linux\OmenCore.Linux.csproj --no-restore` succeeds after restore
- [ ] `build-linux-package.ps1` completes successfully
- [ ] Produced Linux zip includes CLI, GUI payload, version metadata, and checksum output
- [ ] `omencore-cli diagnose` reports systemd availability, service state, config presence, and bundle extraction readiness
- [ ] `daemon --status` reports service/config state and active performance-hold settings
- [ ] `nvidia-powerd` Dynamic Boost guidance appears for known 8D41 / RTX 50-series cap reports when evidence is present

## 10) Windows Packaging

- [ ] `build-installer.ps1` completes successfully
- [ ] Installer launches, installs, and starts app correctly
- [ ] Refreshed wizard bitmap artwork appears correctly in the installer
- [ ] Portable zip launches correctly on a clean test environment
- [ ] Published checksums are present and match artifacts

## 11) Documentation and Release Notes

- [ ] README release/version references are current
- [ ] INSTALL docs and Linux guide are consistent with shipped behavior
- [ ] Hardware validation caveats and pending physical checks are listed in release notes
- [ ] Discord/GitHub issue attributions are preserved for user-reported fixes

## 12) Final Sign-off

- [ ] No open critical blockers remain in `docs/ROADMAP_v3.6.0.md`
- [ ] CPU/RAM measurement evidence is accepted for all required scenarios
- [ ] Linux restored build validation is accepted
- [ ] Physical fan/RGB validation evidence is accepted or explicitly deferred with release-owner sign-off
- [ ] Release artifacts are verified and uploaded
- [ ] Tag/release notes are prepared
