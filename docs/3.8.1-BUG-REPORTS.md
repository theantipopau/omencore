## BUG-3820-001 - Discord: Application Hang Within 10-20s Of Launch (v3.8.1, OMEN 16-xd0010ax / `8BCD`)

**Source:** Discord, OsamaBiden (OMEN 16-xd0010ax) [BEAM], 2026-06-24, posted within hours of the v3.8.1 release.

### Reporter Environment

- HP OMEN by HP Gaming Laptop 16-xd0xxx, ProductId `8BCD`, AMD Ryzen 7 7840HS / Radeon 780M, RTX 4050 Laptop GPU, 16 GB RAM.
- OmenCore `3.8.1.0`. Two consecutive launches both hung and were force-closed by Windows within 10-20 seconds.
- This model's capability profile has `_workerBackedCpuTempOverrideEnabled = true` ("prioritizing worker-backed LHM CPU temperature for accuracy" — see `ModelCapabilityDatabase.cs` notes for `8BCD`), so it calls into the worker-backed temperature path on every monitoring cycle far more aggressively than most other models.

### Reported Behavior

- Two attached session logs (`OmenCore_20260624_121217.log`, `OmenCore_20260624_121344.log`) both start cleanly, reach "You are running the latest version," go through normal profile-switching activity (Quiet -> Performance -> Balanced), then simply stop — no fatal/managed exception in either log.
- Both logs show repeated warnings in the seconds before the cutoff: `🥶 CPU temperature appears frozen at 40.1°C for N readings` and `🥶 GPU temperature appears frozen at 37.0°C for N readings`, plus a `[ThermalSensorProvider] UI-thread ReadSampleAsync timed out (250ms)` warning.
- Windows Event Viewer (Application log, Event ID 1002, Source "Application Hang") confirms: `OmenCore.exe version 3.8.1.0 stopped interacting with Windows and was closed`, `HangType: Cross-process`.

### Root Cause

`HardwareWorkerClient.SendRequestAsync()` — the named-pipe client OmenCore uses to talk to the out-of-process `OmenCore.HardwareWorker.exe` — reused a single `NamedPipeClientStream` across every request with **no serialization and no recovery after a timed-out read**:

1. The client's per-request read timeout (`RequestTimeoutMs` = 2000ms) is shorter than what the worker can occasionally take to respond under load (GC pause, AMD ADL2/NVAPI driver call, etc.).
2. When a read timed out, the client gave up and returned `""`, but the late response — once the worker actually finished — was left sitting, fully framed and unconsumed, in the pipe's kernel receive buffer.
3. The *next* request's read would then consume that stale leftover message instead of its own reply (since the connection was never torn down or resynchronized), permanently shifting every future request/response pair by one. This silently explains the repeated "temperature appears frozen" warnings in both logs — `EnsureCacheFresh()`/`GetCpuTemperature()` kept getting back stale or mismatched data, which the freeze-detector correctly flagged but never recovered from.
4. `WmiBiosMonitor`'s `TryApplyCpuTemperatureFallback()` wraps each of these now-perpetually-failing worker calls in `Task.Run(...).Wait(CpuFallbackReadTimeoutMs)` every monitoring tick (every 2-5s). With the underlying connection permanently desynced, every tick retried and failed the same way, consistent with the escalating thread-pool/responsiveness pressure that Windows ultimately reported as a cross-process Application Hang.

This is consistent with why `8BCD` hit the bug hard enough to hang within seconds: `_workerBackedCpuTempOverrideEnabled` means this model calls into the affected pipe path on *every* monitoring cycle, not just during an authority-mismatch fallback like most models.

### Fix (v3.8.2)

- `HardwareWorkerClient.SendRequestAsync()` now serializes all pipe requests behind a `SemaphoreSlim(1, 1)` and, on any write/read failure or timeout, disposes the pipe handle instead of reusing it. The existing `ShouldRecoverConnection` / `TryConnectToExistingWorkerAsync` / `TryRestartWorkerAsync` recovery path (already used for worker-process-death recovery) now also handles this, establishing a fresh, correctly-synchronized pipe on the next call.
- `WriteAsync`/`FlushAsync` are now covered by the same per-request cancellation token as the read (previously unguarded against hanging indefinitely).
- No fan/thermal control timing, activation, or debounce logic was touched — this is an IPC transport fix only, per this project's standing rule that control-policy changes require physical-hardware evidence.

### Automated Coverage

- `HardwareWorkerClientPipeTests.SendRequestAsync_DisposesPipe_WhenServerNeverResponds_SoNextCallReconnectsInsteadOfReadingStaleData`
- `HardwareWorkerClientPipeTests.SendRequestAsync_SerializesConcurrentCalls_SoEachCallerGetsItsOwnMatchingResponse`
- Full Release test suite re-run clean: 895/895.

### Evidence Still Needed / Hardware Acceptance Criteria

- Reporter confirms v3.8.2 launches and runs without hanging on the same `8BCD` hardware, ideally across several consecutive launches matching the two that previously failed.
- If a hang or repeated "temperature frozen" warning still occurs post-fix, capture a fresh session log plus Event Viewer entry — that would indicate either a second, distinct issue or that the worker itself (not just the pipe client) is genuinely stalling, which would need separate investigation of `OmenCore.HardwareWorker.exe`'s own ADL2/NVAPI call paths.

### Corroborating Evidence: This Is Not `8BCD`-Specific (2026-06-24, v3.8.0 Victus 16-d1xxx Session Log)

A second, independently-submitted v3.8.0 session log (Victus by HP Laptop 16-d1xxx, ProductId `8A25`, RTX 3060) shows the same underlying symptom signature — but far more severe, and on a model with no worker-backed-CPU-temp override:

- Repeated `🥶 CPU temperature appears frozen` warnings recovering only after extremely long stalls: 79s, 412s, 462s, 1900s (~32 min), and **2194s (~36 min)** in a single ~80-minute session.
- The CPU thermal authority repeatedly flip-flops between `ACPI Thermal Zone` and `LHM Fallback`, including at least one flip during an active load spike where the rejected WMI/ACPI reading (80.1C) was *more* plausible than the fallback it switched to (53.0C) — i.e., the worker-backed fallback was returning a stale/wrong value while the system was genuinely hot (`Notification: Temperature warning - CPU at 85°C` fires minutes later).
- Dozens of `External fan reset suspected - Max mode re-applied after sustained drop` warnings cluster in the same windows as the long freezes, consistent with the fan controller also working from stale/desynced telemetry during these stretches.
- The app did not hang or get closed by Windows in this session (it ran for ~80 minutes and shut down cleanly via the tray), so this is evidence of the *same desync mechanism* described above in a degraded-but-surviving form, not a second crash.

This matches the v3.8.2 root cause exactly: a `HardwareWorkerClient` pipe response, once delayed past the client's timeout, is never correctly resynchronized, so the worker-backed CPU temperature (used as a fallback/authority source on essentially every model, not just `8BCD`) keeps returning stale or mismatched data until something forces a reconnect (worker restart, resume-from-sleep recovery, etc.) — which is why recovery here took tens of minutes rather than seconds. The v3.8.2 fix (serialize pipe requests, tear down and reconnect on any read/write failure) addresses this mechanism generally, not just the `8BCD` hang path. No code change beyond what shipped in v3.8.2 is recommended from this log alone; it should be treated as supporting evidence, and a fresh log from this reporter post-3.8.2 would confirm whether freeze durations drop to the bounded few-second range seen elsewhere.

## BUG-3820-002 - Discord: GPU Locked To Base TGP, Dynamic Boost Not Activating In Performance Mode

**Source:** Discord, AcE_CentriC, 2026-06-24. Victus by HP Laptop 16-d1176TX (`8A25` family), RTX 3060 Laptop GPU (base TGP 85W, boost up to 100W), OmenCore v3.8.0. No log attached yet (the report itself flags "Mandatory: Attach Logs" as still outstanding).

### Reported Behavior

- GPU power appears locked at the 85W base TGP in games even while Performance mode is selected; NVIDIA Dynamic Boost never appears to engage to push toward the 100W ceiling.

### Investigation (Code Read Only, No Log Yet)

- OmenCore does not implement or directly toggle NVIDIA Dynamic Boost anywhere in the codebase (no `DynamicBoost` references exist). The only GPU-power-adjacent control OmenCore exposes on this model class is the HP "GPU Power Boost" WMI preset (`SystemControlViewModel.GpuPowerBoostLevel`, Minimum/Medium/Maximum), which influences the TGP ceiling available to the driver/Optimus — it does not directly drive Dynamic Boost itself.
- A separately-submitted v3.8.0 log from what appears to be the same reporter/model (see corroborating evidence above, under BUG-3820-001) shows `GPU Power Boost available via WMI BIOS. Detected: Maximum, User pref: Minimum` at startup, followed immediately by `Startup performance restore is disabled - skipping automatic GPU Power Boost reapply`. That means: the *actual current hardware* GPU Power Boost level at launch was already `Maximum` (good — not locked), and OmenCore's startup-restore safety gate correctly avoided overwriting it. Across the rest of that session, `PerformanceModeService` never references `GpuPowerBoostLevel` at all when applying Quiet/Balanced/Performance profiles, so switching performance modes does not touch the GPU Power Boost preset.
- Conclusion from code alone: there is no code path in this version that would *actively* write a throttled GPU Power Boost level when the user selects Performance mode. If the TGP lock is real and reproducible, the more likely explanations are (a) a genuine NVIDIA driver/Optimus-side Dynamic Boost limitation independent of OmenCore, or (b) some other interaction not yet visible without this reporter's own diagnostic export.

### Evidence Needed Before Any Change

- This reporter's own session log and `core-control-readiness.txt` export, specifically the `GPU Power Boost` detection/apply lines and the GPU OC panel state during a game session where the lock is observed.
- Confirmation of NVIDIA driver version and whether Dynamic Boost is enabled/visible in NVIDIA's own control panel/GeForce Experience independent of OmenCore.
- Whether toggling OmenCore's GPU Power Boost preset to "Maximum" via the Tuning page (manual Apply) changes the observed TGP ceiling at all — if not, this points away from OmenCore entirely.

### Hardware Acceptance Criteria

- Not applicable yet — no code change is proposed. This is logged for tracking only until the requested log is provided.

## BUG-3820-004 - GitHub #146: OMEN 15 `88D2` Stand-By/Fan Management — Fans Stuck At Max Through Lid-Close, BIOS Thermal Shutdown

**Source:** [GitHub #146](https://github.com/theantipopau/omencore/issues/146), nsilveri, 2026-06-25. OMEN Laptop 15-en1xxx, ProductId `88D2` (resolves to the `OMEN by HP Laptop 15z-en100 (2021) AMD` model entry — conservative legacy WMI V1 profile, direct EC writes disabled, GitHub #132), AMD Ryzen 7 5800H + NVIDIA RTX 3070 hybrid GPU, v3.8.0. Log `OmenCore_20260625_194723.log` and a diagnostics export attached.

### Reported Behavior

1. Fans sometimes stay at maximum speed even when temperature is below 50°C.
2. Reported "GPU temperature" is ambiguous (iGPU vs dGPU unclear), and is consistently *higher* than CPU temperature even when the dGPU is confirmed idle — the opposite of what OMEN Gaming Hub showed on the same hardware.
3. **Safety-relevant:** fans were stuck at max, the reporter closed the lid to trigger standby, the laptop did not actually enter low-power standby (screen stayed black but fans kept running), and later powered on to a BIOS message reporting a shutdown due to overheating — while the laptop had been sitting closed inside a backpack.

The attached log does not capture the standby incident itself (it ends mid-session, before the report says the incident occurred), so the root cause below is established from a direct code read of the suspend-handling and fan-control paths, not from this specific log.

### Root Cause (confirmed by code read)

`WmiFanController`'s Max-mode keepalive/reassertion timer (`CountdownExtensionCallback`, the same mechanism behind the "External fan reset suspected — Max mode re-applied" log line seen in other field reports) runs on its own independent `System.Threading.Timer`, separate from `FanService`'s monitor loop and with **no suspend awareness of its own**. It was previously stopped only as a side effect reached partway through a successful `WmiFanController.RestoreAutoControl()` call:

- `RestoreAutoControl()` had an early `if (!IsAvailable) return false;` guard *before* the line that stopped the timer — any transient WMI unavailability skipped stopping the timer entirely, leaving Max mode's reassertion active.
- The timer was only stopped after `ResetFromMaxMode()` ran inside `RestoreAutoControl()`'s try block — though `ResetFromMaxMode()` has its own internal catch-all, any other failure ahead of that line had the same effect.
- `FanService.HandleSystemSuspend()` only called `RestoreAutoControlSerialized()` at all `if (FanWritesAvailable)`, and silently discarded its return value either way — a failed or skipped restore during the suspend transition left the keepalive timer running with no further attempt to stop it, while `FanService`'s own monitor loop correctly paused itself (`_systemSuspendActive`) and Windows began suspending.

Net effect: if Max mode was active when the lid was closed, and the BIOS auto-control restore failed or was skipped for any reason during the brief suspend transition window, the keepalive timer kept reasserting Max fan mode via WMI every ~8 seconds for as long as the process had threads scheduled — directly matching "fans remained at full speed" through lid-close, and is the kind of periodic background hardware I/O that can interfere with a clean Modern Standby/S0ix transition on affected platforms.

### Fix

- `IFanController.StopCountdownExtension()` added as a default no-op interface method (mirrors the existing `IsHoldActive` pattern), overridden in `WmiFanControllerWrapper` to delegate to the real timer. Other backends (OGH, EC, fallback) have no such timer and get the safe no-op.
- `WmiFanController.RestoreAutoControl()` now stops the countdown timer **unconditionally, first**, before the `IsAvailable` check and before any reset-sequence logic — closing the gap for every caller (manual "switch to Auto", preset switching, suspend handling), not just the suspend path.
- `FanService.HandleSystemSuspend()` additionally calls `_fanController.StopCountdownExtension()` directly and unconditionally as well (defense in depth for the case where `FanWritesAvailable` is false and `RestoreAutoControl()` is never even invoked), and now correctly logs whether the BIOS auto-control restore actually succeeded instead of unconditionally claiming success.
- Added `FanServiceSuspendTests.cs` (5 tests) proving the timer is stopped on suspend in every failure mode: restore succeeds, restore returns false, restore throws, fan writes unavailable entirely, and stopping the timer itself throws.

This directly addresses reported behavior #1 and #3. It does not change any fan curve, thermal threshold, or EC-write logic while the system is awake — the change is scoped entirely to suspend-time and restore-to-auto behavior, and strictly *reduces* background WMI write activity rather than adding any.

### Investigated But Not Fixed: GPU Temperature Source (reported behavior #2)

Root-caused but deliberately **not changed** in this patch — this is a fan-curve-input correctness question (`WmiBiosMonitor` evaluates fan curves against `Math.Max(cpuTemp, gpuTemp)`), and changing it carries real regression risk for the majority of NVIDIA-equipped models if done without more care:

- `WmiBiosMonitor`'s GPU temperature arbitration unconditionally prefers the NVIDIA dGPU's NVAPI die-temperature reading over whatever the WMI BIOS itself reports as "GPU temperature," whenever NVAPI is available and returns a positive value — regardless of which GPU is actually active. The comment in code states the rationale ("NVAPI reads directly from the GPU die sensor, higher precision"), which holds when the dGPU is the active GPU, but not on a hybrid AMD+NVIDIA laptop where the log confirms `Detected GPU mode: Integrated GPU Only` (AMD iGPU active, NVIDIA dGPU idle) — NVAPI still returns a nonzero die temperature for the idle dGPU, and that value silently overrides the BIOS's own GPU reading.
- Because fan curves use `Math.Max(cpuTemp, gpuTemp)` as their control temperature, an artificially elevated idle-dGPU reading can plausibly explain reported behavior #1 (fans driven high by a GPU number that doesn't reflect real thermal load) as well as #2 directly.
- A correct fix needs a reliable "is the dGPU actually the active GPU right now" signal wired into `WmiBiosMonitor`'s monitoring loop (the existing `Detected GPU mode` detection runs in a different component, not continuously in this loop), without regressing temperature accuracy on the majority of models where the NVIDIA dGPU genuinely is the primary/active GPU. That is a broader, riskier change than this patch's scope justifies on the evidence of a single report.

### Evidence Needed Before Fixing The GPU Temperature Source

- A second corroborating report/diagnostic export from another AMD iGPU + NVIDIA dGPU hybrid model, ideally with the diagnostics zip (not just the `.log`) so the bounded performance snapshot and core-control-readiness sections can be checked for the actual NVAPI vs. WMI BIOS GPU values side by side.
- Confirmation of what OMEN Gaming Hub actually showed for "GPU temperature" on this exact hardware/workload for direct comparison.

### Hardware Acceptance Criteria

- Reporter (or another affected hybrid-GPU user) confirms fans no longer stay stuck at max independent of temperature, and that lid-close reliably enters standby without a subsequent BIOS overheat-shutdown message, on v3.8.2.
- GPU temperature source fix remains open and untouched pending the evidence above.

## BUG-3820-005 - Discord: OMEN MAX 16 `8D41` (ah0500na) - Nine Reported Issues (Brightness Coupling, Power Automation, RGB, Tuning Locks, Optimizer)

**Reporter:** ACF, Discord, 2026-06-27. OMEN 16 Max ah0500na, ProductId `8D41`, Intel Core Ultra 9 275HX + RTX 5080/5090, OmenCore v3.8.2. Model identity resolves correctly (Exact ProductId, High confidence) — this is not an identity-onboarding report like `MODEL-3810-002`/`MODEL-3810-001`, it is nine distinct behavior complaints on an already-known model.

This model's database entry (`ModelCapabilityDatabase.cs:792-822`) is deliberately conservative: `SupportsFanControlEc = false` and `SupportsFanCurves = false` because "EC registers have completely different layout than legacy OMEN models... causes caps lock blinking panic," WMI/ACPI `platform_profile` only, and a narrow `PerformanceModes = { "Default", "Performance", "Cool" }` list. Several of the nine reports trace back to this intentionally narrow surface rather than to a code defect.

### 1. Changing performance mode changes screen brightness

`PerformanceModeService.Apply()` (`PerformanceModeService.cs:76-277`) never calls any brightness/display API — there is no `SetBrightness`/overdrive call anywhere in the performance-mode path. If brightness visibly changes with mode, it is most likely a firmware-level side effect of the WMI BIOS thermal-policy write this model relies on (`AllowDecoupledWmiThermalPolicyFallback = true`), not something OmenCore is requesting. **Evidence needed:** a session log showing the exact moment of the mode switch, to confirm nothing else (e.g. a display-overdrive toggle bundled into a "Performance" quick action) is involved.

### 2. Battery profile set to "Silent" applies as "Custom" or "Balanced" instead

Likely root cause found: `PowerAutomationService`'s default/global `BatteryPerformanceMode` is `"Silent"` (`PowerAutomationService.cs:58,99`), but this model's `PerformanceModes` list is `{ "Default", "Performance", "Cool" }` (`ModelCapabilityDatabase.cs:807`) — **"Silent" is not one of this model's defined modes at all.** `PerformanceModeService.Apply()` does not validate `mode.Name` against the model's allowed list before applying it (`PerformanceModeService.cs:76-103`), so the name passes through, but whatever UI/sync code resolves a display label for an unrecognized name (most likely `SystemControlViewModel`'s mode-badge resolution, or `GeneralViewModel`'s automatic non-persisting sync path documented in `MODEL-3810-002`) is presumably what falls back to "Custom" or "Balanced". **Not guess-fixed** — the exact substitution point needs the reporter's session log (search for "Performance mode:" / "Applying performance mode" around an AC→battery transition) before changing name-resolution behavior.

### 3. Battery/AC mode not applied at boot — root-caused and fixed

`PowerAutomationService.ApplyCurrentProfile()` (`PowerAutomationService.cs:533-539`) is explicitly commented "useful for initial setup" but had **no caller anywhere in the codebase** — confirmed via full-repo search. `PowerAutomationService` only reacts to the `SystemEvents.PowerModeChanged` event; on a fresh boot, that event hasn't fired yet, so any user with Power Automation enabled kept whatever fan/performance state was last manually set (or last startup-restored) until the next AC↔battery transition, regardless of model. **Fixed:** added a call to `_powerAutomationService.ApplyCurrentProfile()` at the end of `MainViewModel.RestoreSettingsOnStartupAsync()` (`MainViewModel.cs`), after the existing GPU-Power-Boost/fan-preset/TCC-offset startup restores, wrapped in its own try/catch. It runs last so Power Automation (if enabled) has final say over the generic last-state restores, matching the feature's purpose. `ApplyCurrentProfile()` already internally no-ops when Power Automation is disabled, so this is zero-risk for users who don't use the feature.

### 4. RGB: keyboard color doesn't change, light bar changes instead

`KeyboardModelDatabase.cs:462-475` already documents this exact risk for `8D41`: `PreferredMethod = KeyboardMethod.HidPerKey` with `FallbackMethods = { NewWmi2023, ColorTable2020 }`, and the entry's own notes say the per-key USB PID (`0x054E`) is "probable but requires field confirmation" and that the WMI ColorTable fallback "applies zone colors to the light bar" on this chassis if the per-key probe fails. `Services/KeyboardLightingService.cs:501` already emits a status warning to this effect. This reads as the HidPerKey probe failing (wrong/unconfirmed PID) and silently falling through to the light-bar-routed WMI fallback. **Evidence needed:** the reporter's session log line `[HidPerKey] PID 0x????...` (the model's own notes call this out as the confirmation mechanism) to learn the actual USB PID and correct the database entry — already an open, documented gap, not a new defect.

### 5/6/7/8. CPU PL1/PL2 locked, GPU "Power Limit Locked", ~2x battery power draw vs. OGH, Dynamic Boost not functioning

All four are consequences of the same, deliberate gate, not independent bugs:

- Double battery power draw is the believable downstream effect of #5/#6: with neither CPU PL1/PL2 nor GPU TGP reducible below BIOS defaults, the Battery profile cannot lower power consumption the way OGH's own (presumably EC- or vendor-signed-firmware-level) path can.
- Dynamic Boost/PPAB has no dedicated OmenCore toggle at all (grepped `Dynamic.*Boost|PPAB` across the codebase — only fan/WMI plumbing references, no enable/disable control surface), so "not functioning" cannot be an OmenCore regression; it is fully delegated to the NVIDIA driver/firmware.

**Why not fixed:** items 5-8 would require either relaxing the EC-write gate (directly contradicts this model's own documented EC-panic safety rationale, and this project's standing rule against changing safety-critical control code without hardware evidence) or controlling registers OmenCore has no documented write path for. No action taken. If OGH is genuinely achieving lower battery TGP through a path OmenCore doesn't have, that would need to be identified specifically (e.g. via OGH's own diagnostic/WMI trace) before any change is considered.

### 9. Optimizer > Storage & Disk > "Disable Last Access Timestamps" fails

OmenCore's `app.manifest` already requests `requireAdministrator`, so the commonly-suspected "missing elevation" explanation does not apply here — the process is already elevated. The method does correctly capture `ErrorMessage` on a non-zero exit code or exception, but nothing in this file confirms whether that message reaches the Settings UI toggle. **Evidence needed:** the reporter's session log around this action (search for "Last access" or "fsutil"), or the literal error text shown in the UI, before guessing further — the command syntax itself looks correct.

The answer depends on GPU backend and action used:

| Path | Saves named profile | Saves current values | Reapplies at OmenCore startup |
|---|---|---|---|
| NVIDIA NVAPI, Save Profile | Yes | In `GpuOcProfiles` | No; profile save/selection alone does not authorize hardware writes |

## MODEL-3810-002 - GitHub #145 OMEN Slim 16 `8D40` Missing Identity; Battery Care Fails; Performance Profile Resets

**Model identity (addressed).** `8D40` / "OMEN Slim" is a new, thinner chassis line not previously in either database. - The General/Dashboard page's Performance/Balanced/Quiet quick-profile buttons (`GeneralViewModel.ApplyPerformanceProfile/ApplyBalancedProfile/ApplyQuietProfile`), which also drive the `Ctrl+Shift+E` hotkey cycle.
- The tray menu's individual performance-mode items (`MainViewModel.SetPerformanceModeFromTray`).

## BUG-3810-005 - Discord Reports: Fan Spikes To ~5000 RPM "For No Reason" At Idle Temp

**Source:** Discord, 2026-06-23 — mattieswza [LFC] ("my fan randomly went to max speed for no reason"); Mr.Carrot (Victus 16-s0xxx) ("Common problem... Still no fix"); OsamaBiden (OMEN 16-xd0010ax) ("Agreed. It sometimes goes to 5000 rpm for no reason then drops back to idle RPM at idle temp").

### Reporter Environment

- At least two distinct models reported: Victus 16-s0xxx and OMEN 16-xd0010ax (`8BCD` family).
- No diagnostic export attached yet — this entry is triage from the symptom description and source code review, not a confirmed root cause.
- Described as a recurring/"common problem" across multiple users, which points toward shared software behavior rather than one user's hardware fault.

### Investigation And Suspected Mechanism

`FanService.CheckThermalProtection()` has three tiers:

1. **Emergency** (`maxTemp >= 95C`): activates immediately on a single sample, by design, with no debounce. The existing test `ThermalProtection_EmergencyThreshold_RaisesImmediately` explicitly encodes this as the required, intentional behavior — comment: "Emergency... immediate max fans (no debounce, safety critical)."
2. **Warning** (`maxTemp >= _thermalProtectionThreshold`, default 90C): requires 5s of sustained reading before activating (`ThermalActivateDebounceSeconds`).
3. **Release**: requires the temperature to stay below `threshold - 10C` hysteresis for 15s (`ThermalReleaseDebounceSeconds`) before restoring the prior fan state.

`cpuTemp`/`gpuTemp` are passed into this method as raw, unsmoothed per-poll readings with no rolling average or outlier rejection beyond a `[-10C, 150C]` sanity clamp. The reported symptom — fans jump to ~5000 RPM at idle temperature, then drop back to idle RPM after a delay — is consistent with a **single transient or erroneous temperature sample reaching the 95C emergency tier**, which forces 100% fans immediately (tier 1), followed by the temperature reading idle again on the next sample, which then needs the 15s release hysteresis/debounce (tier 3) to elapse before fans drop back down. That elapsed-then-released pattern matches "goes to 5000 RPM... then drops back to idle RPM at idle temp" closely.

### Why This Was Not Guess-Fixed

The zero-debounce emergency activation is a **deliberate, already-tested safety design**, not an oversight — weakening it (e.g. requiring a second consecutive sample before forcing Max) would directly contradict an existing test that encodes the no-debounce requirement as correct, and would slow the response to a genuine thermal emergency by one polling interval. Given this project's standing rule to treat thermal-safety code changes as critical and evidence-gated, this debounce was **intentionally left unchanged** in this pass. The actual question — is this a real (if very brief) thermal event, or a sensor/EC read glitch reporting a wrong value — cannot be answered without seeing the actual cpu/gpu values at the moment of an activation.

### Implemented (Diagnostics Only, No Behavior Change)

- `CheckThermalProtection` now calls `RecordFanCommand(...)` on every Emergency activation, Warning activation, and Release, including the **individual** `cpuTemp` and `gpuTemp` values (not just the combined `maxTemp` that was already logged). These entries are queryable via `FanService.GetCommandHistorySnapshot()` and already flow into existing diagnostic exports (`core-control-readiness.txt` / fan command history) without any new export plumbing.
- This makes the next occurrence diagnosable: if `core-control-readiness.txt` shows an Emergency activation with, say, `cpu=96.0C gpu=41.0C` immediately preceded by idle-range readings and immediately followed by idle-range readings again, that is strong evidence for a single-sample glitch rather than a real thermal event — at which point a targeted fix (e.g. sensor-source filtering, or a very short same-tier confirmation window) could be considered with actual evidence instead of a guess.
- Added `ThermalProtectionTests.ThermalProtection_EmergencyActivation_RecordsBothIndividualReadings`. No existing test changed; all prior thermal-protection tests (including the no-debounce requirement) still pass unmodified.

### Evidence Needed Before Any Behavior Change

- A diagnostic export (`core-control-readiness.txt` / fan command history) captured at or shortly after a spike, showing the `ThermalProtection.Emergency` entry's individual cpu/gpu values and the samples immediately before/after it.
- Whether the spike correlates with a specific monitoring source (WMI vs worker-backed LibreHardwareMonitor) — i.e. is one sensor path more prone to this than another.
- Whether the spike correlates with any system event (resume from sleep, GPU driver activity, a brief real CPU boost under a benchmark/game launch) rather than being purely random.
- Confirmation of OmenCore version/build and whether `8BCD`'s already-documented degraded-WMI-control behavior (Linux #137 territory, but these are Windows reports) plays a role.

### Candidate Code Areas (For A Future, Evidence-Backed Fix)

- `src/OmenCoreApp/Services/FanService.cs` (`CheckThermalProtection`, done for diagnostics; activation timing intentionally untouched)
- `src/OmenCoreApp/Hardware/ThermalSensorProvider.cs` and worker-backed sensor paths, if evidence points to a specific sensor source producing the bad sample
- `src/OmenCoreApp.Tests/Services/ThermalProtectionTests.cs`

### Hardware Acceptance Criteria

- Not applicable yet — this item is in the evidence-gathering stage. No fix should be attempted until a diagnostic export from an affected user is reviewed.
