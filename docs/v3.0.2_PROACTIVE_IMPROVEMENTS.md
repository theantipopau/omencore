## Pattern 1: Extend Sensor Freeze Detection to GPU Temperature

### Issue
- CPU temp freeze detection added in v3.0.2 main fixes
- GPU temperature sensor could have the same freeze issue
- Currently unmonitored, could cause misleading thermal data on AMD systems

### Solution Implemented ✅

**File:** `src/OmenCoreApp/Hardware/WmiBiosMonitor.cs`

Added GPU temperature freeze detection parallel to CPU:
- `_lastGpuTempReading` — track previous GPU temp
- `_consecutiveIdenticalGpuTempReads` — count identical readings
- `_gpuTempFrozen` — status flag
- Detect freeze when GPU maintained same temp for 30+ readings
- Log warning/recovery same as CPU: `🥶 GPU temperature appears frozen at...`

**Scope:** WMI BIOS GPU temperature path (most common source on OMEN laptops)

**Why This Matters:**
- GPU temp is used for thermal protection and fan control curve evaluation
- Frozen GPU temp → fan curve won't respond to actual GPU load increase
- Silent failure on AMD models would cause thermal issues

## Pattern 2: Centralize Driver Post-Installation State Detection

### Issue
- `PawnIOMsrAccess.IsPawnIOInstalled()` method needed for reboot detection
- Same method needed in `PawnIOEcAccess` and `PawnIOCpuTemp` (both PawnIO modules)
- WinRing0 also needs post-install detection
- Code duplication across multiple classes

### Solution Implemented ✅

**File:** `src/OmenCoreApp/Hardware/DriverInitializationHelper.cs` (NEW)

Created centralized utility class:

```csharp
public static class DriverInitializationHelper
{
    public static bool IsPawnIOInstalled()  // Unified detection
    public static bool IsWinRing0Installed()  // Future proofing
    public static string? GetPostInstallationRebootWarning(...)  // User messages
}
```

**Benefits:**
- Single source of truth for driver detection logic
- Future-proof for other drivers (e.g., LibreHardwareMonitor, DPC Latency Checker)
- Easier to test and maintain
- Consistent messaging across all driver initialization paths

**Updated:**
- `PawnIOMsrAccess.IsPawnIOInstalled()` now delegates to helper
- Maintains backward compatibility (public API unchanged)
- Other PawnIO classes can use this when updated

## Pattern 3: Identified But Deferred - EC Write Debouncing on AMD

### Observation

During analysis, found `DutyDeduplicationSeconds = 15.0` in `FanController.cs`:
- Prevents EC write hammering
- Safe on Intel models
- **Might be too aggressive on AMD OMEN 16 xd0xxx**

### Current Status: MONITORED
- Not changed yet (need community validation that countdown extension fix is sufficient)
- If users report fans still bouncing after v3.0.2 release, this is the next target
- Comments added in WmiFanController identifying this for future tuning

### Why Deferred:
- Countdown extension (800ms) is already very aggressive
- Changing deduplication window needs wider testing (could affect all models)
- Better to release current status and get telemetry
